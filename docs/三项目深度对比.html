🔬 关键机制详解

🧬 两步链式思考 Ingest nashsu
不是一步到位让 LLM 直接写 wiki，而是分两步：
  • Step 1 分析：提取实体、概念、论点、与现有 wiki 的关联、矛盾点
  • Step 2 生成：基于分析结果写摘要页、实体页、概念页、更新索引
类比：先列提纲再写文章，质量远超直接写。
🔥 Hot Cache 机制 claude-obsidian
wiki/hot.md 是约 500 词的近期上下文缓存，通过 Hook 体系实现：
  • SessionStart：自动读取 hot.md → 新 session 秒有上下文
  • Stop：检测 wiki 变更 → 提示更新 hot cache
解决了"AI 每次醒来都是失忆状态"的痛点。
📊 四信号知识图谱 nashsu
不靠 LLM 自由发挥连线，而是用四个可量化信号评分：
  • 直接链接 ×3.0 — [[wikilinks]] 互联
  • 来源重叠 ×4.0 — 共享相同原始素材（权重最高）
  • Adamic-Adar ×1.5 — 共同邻居加权
  • 类型亲和 ×1.0 — 同类型页面加分
再用 Louvain 算法自动检测知识社区/聚类。
🏥 10 类 Lint 健康检查 claude-obsidian
最完整的知识库健康维护体系：
  • 孤立页面 · 死链接 · 陈旧声明
  • 缺失页面 · 缺失交叉引用
  • frontmatter 缺失 · 空白章节
  • 陈旧索引条目 · 地址验证
  • 语义去重（用 embedding 检测重复内容）
🔍 Deep Research 主动研究 nashsu
不只消化已有素材，还主动上网填补知识空白：
  • LLM 读 overview + purpose → 生成研究主题和搜索词
  • 用户确认后，通过 Tavily/SerpApi/SearXNG 搜索
  • LLM 综合结果写入研究页
  • 研究结果自动 ingest → 提取新的实体/概念
也能从图谱洞察触发：孤立节点、稀疏社区、桥接节点。
⚡ 三层分离 + Git 版本化 gbrain
架构设计最清晰的一个：
  • Brain Repo：纯 Markdown 文件 + Git 管理 = 可回溯
  • Retrieval：Postgres 做混合检索（应对大规模）
  • MCP Server：标准接口，Agent 可读/写/推理
gbrain features --auto-fix 自动诊断并修复问题。
⚔️ 矛盾双向标注 claude-obsidian
新素材与已有知识冲突时，不是静默覆盖：
  • 在两个冲突页面都添加 [!contradiction] callout
  • callout 内包含对方页面的链接和冲突内容
  • 人类可以后续裁决哪个正确
其他项目最多只是标记，不做双向关联。
🎯 purpose.md — Wiki 的灵魂 nashsu
每个 Wiki 项目都有一个 purpose.md：
  • 定义目标：这个知识库是用来做什么的
  • 关键问题：我最想回答的 3-5 个问题
  • 研究范围：什么该收、什么不收
  • 演进论点：我目前的核心观点（允许随证据变化）
LLM 在每次 ingest 和 query 时都读取，保证行为一致。
