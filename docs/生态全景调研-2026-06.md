## 二、重大平台级变化（4月调研后新增）

### 2.1 Anthropic "Dreaming" — Agent 记忆巩固（2026-05-06）

**最重要的行业变化。** Anthropic 在 Code with Claude 大会上发布 Dreaming 功能：

- 调度式后台进程，Agent 在闲置时"做梦"
- 回顾过去的 session，提取模式，策展记忆
- Harvey 法律 AI 公司报告开启后任务完成率提升 ~6x
- 本质上是 TideMind "活记忆"理念的官方实现

**对 Stitch 的影响**：TideMind 的"记忆代谢"不再是实验性思路，而是被 Anthropic 验证的方向。但 Dreaming 目前仅限 Managed Agents，本地 Claude Code 暂未支持。

## 五、关键新思路与趋势

### 5.1 "LLM Wiki 缺什么" — 社区共识（5 个扩展）

来源: [ghelburlabs Substack](https://ghelburlabs.substack.com/p/i-rebuilt-karpathys-llm-wiki-heres) + [penfieldlabs](https://penfieldlabs.substack.com/p/what-karpathys-llm-wiki-is-missing)

1. **Write-back** — Query 的好答案应存回 Wiki
2. **Automatic Reconciliation** — 新旧知识冲突时自动调和
3. **Unsolicited Synthesis** — LLM 主动综合，不等你问
4. **Scheduled Agents** — 定期跑 lint/synthesis/维护
5. **AI-first Notes** — 笔记为 AI 检索优化，不是为人类阅读优化

### 5.2 从"编译"到"可查询图谱"

- Graphify: 文件夹 → 知识图谱 → 查询图谱（而非读文件）
- obsidian-graph: pgvector 语义搜索层
- Project Synapse: Neo4j + Obsidian Markdown wiki 混合
- Graph Explorer: LLM Wiki 需要可视化层来发现聚类和缺口

### 5.3 从"个人"到"团队"

- Beever Atlas: 团队聊天 → Living Wiki（最重要的方向演进）
- Compound Engineering: 团队工程知识复利
- 企业级需求：审计、权限、多人协作

### 5.4 跨工具统一记忆

- Supermemory: Claude Code + OpenClaw + OpenCode + Hermes
- TideMind: MCP 协议连接所有 AI 工具
- Remember.md: wikilinks 结构化跨 session 记忆
- 痛点不再是"没有知识库"，而是"记忆散落在 10 个工具里"

### 5.5 Eval/质量保证

- [I Shipped the Eval Harness Instead](https://medium.com/@rex_colon.md/i-tried-to-build-karpathys-llm-wiki-i-shipped-the-eval-harness-instead-15b174870895)
- 从"能跑就行"到"能验证 ingest 质量"
- 置信度标注 + 矛盾检测 + lint 分数

### 9.1 GBrain — Garry Tan (YC CEO)

- **GitHub**: [garrytan/gbrain](https://github.com/garrytan/gbrain) ⭐ ~14,000
- **发布**：2026-04-05（24小时 5K stars）
- **定位**：Opinionated OpenClaw/Hermes Agent Brain
- **架构**：三层
  - Brain Repo — git 版本化的 Markdown 文件
  - GBrain Retrieval — Postgres 混合检索（BM25 + 向量）
  - MCP Server — Agent 接口
- **核心特色**：
  - Markdown-first 知识图谱，plain-text → 自动连线
  - AI agent 可读/写/推理
  - `gbrain features --json --auto-fix` — 自动扫描未使用功能并修复
  - 自带健康检查（dead links、missing embeddings）
- **许可**：MIT
- **生态对比**：[GBrain vs Hindsight vs Mem0 vs Zep](https://vectorize.io/articles/gbrain-vs-hindsight-vs-mem0-vs-zep)
- **对 Stitch 价值**：架构参考（git-versioned brain + hybrid retrieval + MCP），但绑定 OpenClaw 生态较深

### 9.2 sdyckjq-lab/llm-wiki-skill

- **GitHub**: [sdyckjq-lab/llm-wiki-skill](https://github.com/sdyckjq-lab/llm-wiki-skill)
- **定位**：基于 Karpathy 方法论的个人知识库 Skill，支持多平台
- **多平台**：Claude Code / Codex / OpenClaw

- **关联**：与 `lewislulu/llm-wiki-skill`（⭐731）同系列/fork，中文生态

### 9.3 yifanfeng97/Hyper-Extract（iMoonLab 清华体系）

- **GitHub**: 未直接找到独立 repo，yifanfeng97 = iMoonLab
- **关联项目**：
  - [iMoonLab/Hyper-RAG](https://github.com/iMoonLab/Hyper-RAG) — 超图驱动 RAG 对抗 LLM 幻觉
  - [iMoonLab/LLM4Hypergraph](https://github.com/iMoonLab/LLM4Hypergraph) — ICLR 2025
- **学术论文**：
  - [Skill-Driven Knowledge Extractor for Knowledge Hypergraph Generation](https://arxiv.org/abs/2602.19543)（2026-02）
  - Hyper-FM: Hypergraph Foundation Model（2025-03）
- **核心思想**：
  - 知识超图（hypergraph）> 传统二元知识图谱 — 能表达 n-ary 复杂事实
  - 自适应技能获取模块 + 稳定性反馈循环 + Global Skill Library
  - 非结构化文本 → 结构化知识（图 + 超图 + 时空抽取）— "one command"
- **对 Stitch 价值**：学术前沿方法，适合需要复杂关系抽取的场景。工程化程度待验证，观望为主

### 9.4 opendatalab/MinerU-Document-Explorer ⭐⭐⭐（强烈推荐）

- **GitHub**: [opendatalab/MinerU-Document-Explorer](https://github.com/opendatalab/MinerU-Document-Explorer)
- **母项目**：[opendatalab/MinerU](https://github.com/opendatalab/MinerU) — 成熟文档解析工具
- **定位**：Agent-native 知识引擎
- **输入格式**：PDF / DOCX / PPTX / Markdown
- **检索**：Hybrid search — BM25 + 向量搜索 + LLM reranking
- **核心功能**：
  - Document indexing（文档索引）
  - Wiki organization（wiki 组织）
  - Fast retrieval（快速检索）
  - Deep reading（深度阅读）
- **接口**：
  - MCP Server（已上架 [LobeHub MCP](https://lobehub.com/mcp/opendatalab-mineru-document-explorer)）
  - Agent Skill（`skills/mineru-document-explorer/SKILL.md`）
  - Demo 目录
- **MinerU 母项目能力**：
  - 复杂布局解析、扫描页、手写内容
  - PDF/Image/DOCX/PPTX/XLSX → Markdown/JSON
  - [官方文档](https://opendatalab.github.io/MinerU/)
- **对 Stitch 价值**：**解决文档 ingest 最大痛点**。MinerU 本身是非常成熟的文档解析器，Document Explorer 在此基础上加了 wiki organization + hybrid retrieval。直接作为 Stitch 的文档输入层

### 9.5 ussumant/llm-wiki-compiler ⭐⭐⭐（编译核心参考）

- **GitHub**: [ussumant/llm-wiki-compiler](https://github.com/ussumant/llm-wiki-compiler)
- **文档站**：[Mintlify 文档](https://saydo-5cd0e3d7.mintlify.app/)
- **Skill 市场**：[skillsmp.com](https://skillsmp.com/skills/ussumant-llm-wiki-compiler-plugin-skills-wiki-compiler-skill-md)
- **定位**：Claude Code plugin — 散落 markdown → topic-based wiki
- **核心特色**：
  - **两阶段编译 pipeline**（分析→生成）
  - **增量变更检测**（SHA hash，跳过未变化文件）
  - **Claim-level provenance**（声明级来源追溯）— 每条知识可追溯到原始素材
  - 可逆编译
  - 支持 knowledge mode（markdown）+ code mode
- **工程化评价**：被 Medium 深度评测为"第一个把 LLM Wiki 当做**真正的编译器问题**来处理的实现"
- **深度解析**：[Compiling knowledge, not retrieving it](https://medium.com/@roanmonteiro/compiling-knowledge-not-retrieving-it-a-hands-on-deep-dive-into-llm-wiki-compiler-a495523a8085)
- **相关学术**：[Wiki-memory: Compile, Evaluate, Refine — Iterative Knowledge Compilation for LLM Wiki Systems](https://arxiv.org/html/2605.07068v1)（2026-05 arXiv）
- **对 Stitch 价值**：**编译核心的最佳工程参考**。两阶段 pipeline + hash 增量 + claim provenance 正是方案 B 需要的核心算法

## 十、社区声音总结（linux.do 48 条讨论精华）

### 共识

1. **规模瓶颈是真实痛点** — sources < 100 可以纯 LLM Wiki（上下文装得下），超过后必须有索引/RAG 层
2. **方向对，但不必全盘照搬** — Karpathy "总提大思想解决小问题"，项目扩大化后有些理论失效
3. **graphify 对代码好但对文档一般** — 多人反馈
4. **GitHub 上大多数实现太浅** — "要不做的太浅，要不太工程化了"

### 有价值的实战案例

- 占卜知识库：用 Karpathy 思想做垂直领域（skill 自动接收→解卦→存档→反馈→纠正→agent 隔离回测→公理提取），证明非代码场景可行
- 文档/PDF 场景：graphify 第一阶段（代码抽取）可剥离，只保留第二阶段（编译），效果还行但不够好

### 争议点

- "RAG 上下文越多→幻觉越大" — 有一定道理但非绝对
- "企业级几十万行代码，这些观点没一个涉及深水区" — 确实适用于个人/小项目
- "等 AI 能自己写蓝图自己审计才有用" — 太理想化，当前实用价值已在
