深层技术对比

nashsu/llm_wiki — VS — claude-obsidian

从 Ingest 流水线到检索策略、从 Schema 设计到规模瓶颈 —— 七个维度逐层拆解

① Ingest 编译流水线

一个源文件从输入到编译为 Wiki 页面的完整流程。这是两个项目最核心的差异。

nashsu/llm_wiki — 两步链式思考（2-4 次 LLM 调用）
SHA256 哈希检查 — 对比 .llm-wiki/ingest-cache.json，重复则跳过
PDF 图片提取 — Rust pdf-extract 提取嵌入图片，视觉 LLM 生成描述
长文档分块 — 超过 300K 字符则切块（12K-60K），各自分析后合并
第一次 LLM 调用：分析 — 读源文件 + index.md + purpose.md → 输出结构化分析（实体/概念/论点/矛盾/建议）
第二次 LLM 调用：生成 — 基于分析结果，用 ---FILE: path--- 格式批量产出 wiki 页面
原子写入 — 用 mutex 锁保护，批量写入文件系统
向量嵌入（可选）— 新页面自动切块嵌入 LanceDB
第三次 LLM 调用：链接增强（可选）— 读页面+索引，返回 JSON {links: [{term, target}]}
Review 清理 — 自动解决已满足条件的 review items
冲突处理
分析阶段明确要求"与现有 wiki 是否矛盾？"→ 生成阶段输出 ---REVIEW: contradiction--- 块 → page-merge.ts 合并已有页面（不覆盖）
claude-obsidian — 单会话连续执行（1 次 LLM 会话）
MD5 哈希检查 — 对比 .raw/.manifest.json，重复则跳过
完整阅读源文件 — Skill 指令："Do not skim."
讨论步骤（交互式）— 询问用户"强调什么？颗粒度？" 可跳过
创建来源摘要页 → wiki/sources/
创建/更新实体页 — 人物、组织、产品、仓库
创建/更新概念页 — 重要想法、框架
更新领域页 + 子索引 _index.md
更新 overview.md（如大局改变）
更新 index.md + hot.md + log.md
矛盾检查 — 在两个冲突页添加 [!contradiction] callout
冲突处理
Claude 直接在两个页面都写入 > [!contradiction] callout，包含对方链接和冲突内容。永远不会静默覆盖旧信息。
核心差异总结
nashsu：程序化编排，LLM 输出严格格式化（---FILE:--- 块），代码解析后批量写入。可重复、可测试、可缓存。
claude-obsidian：LLM 即执行器，Claude Code 直接调用 Read/Write/Edit 工具操作文件。更灵活但不可重复。

② 页面结构与 Schema 设计

nashsu/llm_wiki

页面类型（9 种）

source · entity · concept · comparison · query · synthesis · thesis · methodology · finding

Frontmatter 格式

type: entity title: "Andrej Karpathy" created: 2026-04-10 updated: 2026-06-03 tags: [ai-researcher, openai] related: [llm-wiki, vibe-coding] # 裸 slug，无 [[]] sources: ["karpathy-gist.md"] # 原始文件名

特点

  • related 用裸 kebab-case slug（无双括号、无路径）
  • 页面类型更多（thesis/methodology/finding/synthesis）
  • 5 种场景模板预配置 purpose.md + schema.md
  • 用户可在 schema.md 自定义页面类型和目录
claude-obsidian

页面类型（8 种）

source · entity · concept · domain · comparison · question · overview · meta

Frontmatter 格式

type: entity title: "Andrej Karpathy" created: 2026-04-07 updated: 2026-06-03 tags: - ai-researcher - openai status: mature # seed/developing/mature/evergreen related: - "[[LLM Wiki]]" # 完整 wikilink 格式 sources: - "[[.raw/articles/karpathy-gist.md]]" entity_type: person # 类型专属字段 role: "AI researcher, OpenAI co-founder" confidence: high

特点

  • related 用完整 [[wikilink]] 格式（Obsidian Properties 兼容）
  • 有 status 生命周期（seed→developing→mature→evergreen）
  • 有 confidence（high/medium/low）标注知识可信度
  • 类型专属字段更丰富（entity_type/complexity/verdict 等）
  • 4 种方法论模式（LYT/PARA/Zettelkasten/通用）
  • 6 种用途模式（Website/GitHub/Business/Personal/Research/Book）

③ 查询与检索策略

nashsu — 四阶段检索管线
分词搜索 — 英文词分割+停用词；中文 CJK bigram。标题匹配 +10 分
向量语义搜索（可选 LanceDB）— chunk 级嵌入，余弦相似度，RRF 融合
图谱扩展 — 搜索结果为种子，四信号模型 2-hop 扩展（衰减）
预算控制 + 组装 — 50% wiki 页面 / 5% 索引 / 15% 保留 / 30% 历史+系统
Token 预算管理
显式 context-budget.ts 分配器。可配 4K-1M token 窗口。按 relevance 分数排序页面，逐页加入直到预算耗尽。每页截断上限 = 30% × pageBudget（最低 5K 字符）。
claude-obsidian — 三模式 + 可选混合检索

快速模式（~1,500 tokens）

只读 hot.md → 如果能回答就直接回答 → 否则扫 index.md

标准模式（~3,000 tokens）

hot.md → index.md → 读 3-5 个相关页面 → 跟随 wikilinks 深度 2 → 综合

深度模式（~8,000+ tokens）

读所有相关页面（不跳过）→ 提供网络搜索 → 结果存回 wiki

wiki-retrieve（v1.7 可选升级）

BM25 top-20 → 余弦重排序 → top-5。三层上下文前缀：Anthropic API（Haiku）/ claude CLI / 合成免费层。

Token 预算管理
无显式分配器。依赖 Claude Code 原生上下文管理 + Skill 指令约束（"读 3-5 页"）。Hot Cache（~500 词）是核心省 token 手段。

④ 维护与 Lint 健康检查

nashsu — 结构化检查 + LLM 语义分析

代码级检查（无 LLM）

  • 孤立页面（0 个入站链接）
  • 断开链接（指向不存在页面）
  • 无出站链接（不引用任何页面）

LLM 辅助

  • dedup.ts — 三阶段去重（提取摘要→LLM 检测分组→合并+重写链接）
  • runSemanticLint — LLM 做深层语义分析
  • sweep-reviews.ts — 自动清理已满足条件的 review items
  • source-lifecycle.ts — 级联删除（删源→清实体→清索引→清死链）

触发方式

手动触发（Lint 面板按钮）。无自动/定时 lint。

claude-obsidian — 10 类全覆盖检查
  • ① 孤立页面 — 无入站链接
  • ② 死链接 — 指向不存在的页面
  • ③ 陈旧声明 — 被更新来源矛盾的旧断言
  • ④ 缺失页面 — 多处提到但没有独立页面的概念
  • ⑤ 缺失交叉引用 — 提到实体但未添加链接
  • ⑥ Frontmatter 缺失字段
  • ⑦ 空白章节
  • ⑧ 陈旧索引条目
  • ⑨ 地址验证（DragonScale，可选）
  • ⑩ 语义去重 — ollama nomic-embed-text 余弦相似度 ≥0.90 报错，0.80-0.90 待审

输出

wiki/meta/lint-report-YYYY-MM-DD.md，分级别（可自动修复 / 需人工审核）

触发方式

手动触发（"lint the wiki"）。建议每 10-15 次 ingest 后运行一次。

⑤ 交叉链接策略

nashsu — 三重机制 + 量化评分

机制一：Ingest 时 LLM 生成

生成 prompt 指示 LLM 在页面正文使用 [[wikilink]]，在 frontmatter 填写 related 字段。

机制二：Post-ingest 链接增强

enrich-wikilinks.ts — 额外 LLM 调用，返回 JSON → 代码做首次出现替换。每个目标只链接一次，只链接索引中已有的页面。

机制三：四信号图谱发现隐式关系

  • 直接链接 ×3.0
  • 来源重叠 ×4.0（最高权重！共享原始素材 = 强关联）
  • Adamic-Adar ×1.5（共同邻居，按度数加权）
  • 类型亲和 ×1.0（含跨类型亲和矩阵）
防过度/不足链接
过度：每个目标只链接首次出现，只链接已有页面。
不足：post-ingest enrichment 专门补漏。
隐式关系：图谱信号发现非显式链接的关联（如共享来源但未互链的页面）。
claude-obsidian — LLM 判断 + Lint 兜底

机制一：Ingest 时 LLM 判断

Skill 指令："在正文中对所有已知实体/概念使用 [[wikilink]]。" Claude 根据阅读 index.md 和现有页面做判断。

机制二：Lint 检查 #5 兜底

"缺失交叉引用"检查 — 发现正文提到但未链接的实体，自动修复可添加链接。

无量化评分系统

完全依赖 LLM 的自然语言理解判断什么该链接。没有权重、阈值或评分矩阵。

防过度/不足链接
不足：Lint #5 兜底检测。
过度：无显式防护，依赖 LLM 判断力 + 上下文约束（"只读 3-5 页"间接限制链接密度）。
无隐式关系发现机制。

⑦ 局限性与已知问题

nashsu/llm_wiki 的问题

  • PDF ingest 不稳定 — 同一 PDF 时成时败
  • 批量 ingest 质量退化 — 超过 ~50 个源后 index/overview 停止更新
  • 中文文件名编码 — 被 URL 编码
  • Deep Research 并发冲突 — 多任务覆盖输出
  • 安全顾虑 — 硬编码 API 端口，缺少 CSP，LLM 写入未校验
  • 单一自定义模型 — 只能配一个 custom endpoint
  • 无 Hot Cache — 每次 session 无持久上下文
  • 无跨项目引用
  • 桌面 Only — 无 CLI/终端工作流
  • GPL-3.0 — 传染性协议，商用需注意

claude-obsidian 的问题

  • Hook 兼容性 — Claude Code v2.1.140+ 的 prompt-type hook 会断
  • Lint 即兴执行 — 核心检查 1-7 没有固定数据采集方法，Claude 每次跑法不同
  • 无结构化搜索后端 — 依赖 Grep/Glob（wiki-retrieve 可选但需 Python+ollama）
  • 无独立桌面应用 — 必须有 Claude Code 订阅
  • 无图谱可视化 — 依赖 Obsidian 原生 graph view
  • 无多格式文档支持 — 不能直接解析 PDF/DOCX/PPTX
  • 无 Chrome 剪藏扩展
  • 无社区检测/图谱洞察
  • 无持久 ingest 队列（无崩溃恢复）
  • wiki-retrieve 规模上限 — >500 页警告，>5000 页失败
规模相关nashsuclaude-obsidian
Ingest 队列持久化队列 + 崩溃恢复无（单次 session）
搜索规模LanceDB 向量 + sigma.js 数百节点BM25 O(N²) in Python，>5000 页失败
并行 Ingest串行（防 LLM 冲突）Sub-agents 可并行（文件锁）
长文档自动分块 12K-60K + 合并依赖 Claude 原生上下文窗口
图片处理PDF 图片提取 + 视觉 LLM 描述Claude 原生视觉（无批量处理）

🎯 Stitch 应该怎么选

不是二选一，而是取各自最强的部分组合。两个项目的强项恰好互补。

从 nashsu 拿编译引擎

两步链式思考 · SHA256 增量 · 四信号图谱 · 长文档分块 · Deep Research · purpose.md 目标约束 · Chrome Web Clipper · 显式 token 预算控制 · 多 LLM 提供商支持

从 claude-obsidian 拿维护体系

Hot Cache 秒加载 · 10 类 Lint · [!contradiction] 双向标注 · status 生命周期 · confidence 置信度 · 方法论切换 · Hook 四钩子 · Auto-commit · 交互式讨论步骤 · 隐私同意门

两者都有但实现不同，取更优的

去重：nashsu 的 dedup.ts（三阶段）vs claude-obsidian 的语义 tiling → 取 nashsu（更系统化）
矛盾处理：nashsu 的 review item vs claude-obsidian 的 callout → 取 claude-obsidian（更直观）
MCP：两者都有 → 取 nashsu（更完整的 tool 集）
Schema：nashsu 9 类型 vs claude-obsidian 8 类型+lifecycle → 取 claude-obsidian（lifecycle 有价值）
