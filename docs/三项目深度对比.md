## 二、nashsu/llm_wiki 详解

### 基本信息
- **仓库**: https://github.com/nashsu/llm_wiki
- **版本**: v0.4.17（2026-06-03，两个月发了 37 个版本）
- **技术栈**: Tauri v2 (Rust) + React 19 + TypeScript + Vite
- **LLM 支持**: OpenAI / Anthropic / Google / Ollama / Azure / 自定义端点

### 架构亮点

**三层 + Purpose 层**：
```
raw/sources/  →  wiki/ (编译层)  →  查询/输出
     ↑                              ↑
  purpose.md (Wiki的灵魂：目标、关键问题、研究范围、演进论点)
```

**两步链式思考 Ingest**：
1. **分析阶段**：提取实体、概念、论点、与现有 wiki 的连接、矛盾和张力
2. **生成阶段**：基于分析结果产出摘要页、实体页、概念页、更新索引

**四信号知识图谱**：

| 信号 | 权重 | 含义 |
|------|------|------|
| 直接链接 | ×3.0 | [[wikilinks]] 互联 |
| 来源重叠 | ×4.0 | 共享相同 raw source |
| Adamic-Adar | ×1.5 | 共同邻居（按邻居度数加权） |
| 类型亲和 | ×1.0 | 同类型页面加分 |

**Deep Research**（主动研究）：
1. LLM 读取 overview + purpose → 生成研究主题和搜索查询
2. 用户确认（可编辑查询）
3. 通过 Tavily/SerpApi/SearXNG/AnyTXT 搜索
4. LLM 综合结果写入研究页 → 自动 ingest 提取实体/概念

**Chrome Web Clipper**：
- Manifest V3 扩展，用 Readability.js 提取 + Turndown.js 转 Markdown
- 本地 HTTP API（端口 19827）通信
- 支持多项目选择，剪切后自动触发 ingest

### 目录结构（Wiki 项目内）
```
my-wiki/
├── purpose.md          # Wiki 灵魂
├── schema.md           # 结构规则
├── raw/sources/        # 不可变原始文档
├── wiki/
│   ├── index.md        # 内容目录
│   ├── log.md          # 操作历史
│   ├── overview.md     # 全局摘要（自动更新）
│   ├── entities/       # 人物、组织、产品
│   ├── concepts/       # 理论、方法
│   ├── sources/        # 来源摘要
│   ├── queries/        # 保存的问答和研究
│   ├── synthesis/      # 跨来源综合分析
│   └── comparisons/    # 对比分析
├── .obsidian/          # 自动生成的 Obsidian 配置
└── .llm-wiki/          # 应用配置
```

### 独特优势
- **场景模板**：研究/阅读/个人成长/商业/通用（预配置 purpose + schema）
- **向量搜索**：可选 LanceDB，召回率从 58.2% 提升到 71.4%
- **上下文预算控制**：可配 4K-1M tokens（60% wiki / 20% 历史 / 5% 索引 / 15% 系统）
- **Obsidian 兼容**：wiki 目录就是标准 Obsidian vault
- **MCP Server**：v0.4.17 新增，可被其他 Agent 调用
- **多模态**：PDF 图片提取 + 视觉模型生成描述

## 三、AgriciDaniel/claude-obsidian 详解

### 基本信息
- **仓库**: https://github.com/AgriciDaniel/claude-obsidian
- **版本**: v1.9.2（2026-05-27）
- **形态**: Claude Code 插件（不是 Obsidian 插件！在 CLI 运行，输出到 Obsidian 库）
- **协议**: MIT

### 核心理念
> "Wiki 是产品，Chat 只是接口。知识像复利一样积累。"

### 自组织机制

**Ingest 流程**（一个源文件触达 8-15 个 wiki 页）：
1. 读取源文件
2. 创建来源摘要页 → `wiki/sources/`
3. 创建/更新实体页 → `wiki/entities/`
4. 创建/更新概念页 → `wiki/concepts/`
5. 更新领域页和子索引
6. 更新 `wiki/index.md`（主目录）
7. 更新 `wiki/hot.md`（500 词近期上下文缓存）
8. 追加 `wiki/log.md`
9. 标记矛盾（`[!contradiction]` callout）
10. 添加双向 wikilinks

**Hot Cache 机制**：
- `wiki/hot.md` 是 ~500 词的近期上下文缓存
- SessionStart hook 自动读取 → 每次新 session 立即有完整上下文
- Stop hook 检测 wiki 变更 → 提示更新 hot cache

**自动提交**：PostToolUse hook 在每次 Write/Edit 后自动 git commit

### 15 个 Skills

| Skill | 功能 |
|-------|------|
| wiki | 编排器 + 引用资料 |
| wiki-ingest | 单文件/批量/URL/图片 ingest |
| wiki-query | 从 vault 回答问题 |
| wiki-lint | 健康检查（10 类） |
| wiki-cli | Obsidian CLI 传输层 |
| wiki-retrieve | 混合检索（BM25 + 上下文前缀 + 余弦重排序） |
| wiki-mode | 方法论路由（LYT/PARA/Zettelkasten/通用） |
| wiki-fold | 日志折叠（DragonScale） |
| save | 对话存为 wiki 笔记 |
| autoresearch | 自主多轮网络研究 |
| canvas | 可视化层（图片/PDF/笔记画布） |
| defuddle | 网页提取封装 |
| obsidian-bases | Bases 数据库参考 |
| obsidian-markdown | Obsidian Markdown 参考 |
| think | 10 原则思考框架 |

### Lint 检查（10 类）
1. 孤立页面
2. 死链接
3. 陈旧声明
4. 缺失页面
5. 缺失交叉引用
6. frontmatter 缺失
7. 空白章节
8. 陈旧索引条目
9. 地址验证
10. 语义去重（embedding 相似度检测）

### 方法论模式（四选一）
- **Generic**：通用模式
- **LYT**：Linking Your Thinking（Nick Milo）
- **PARA**：Projects/Areas/Resources/Archives（Tiago Forte）
- **Zettelkasten**：卡片盒笔记法

### Schema 设计

每页必须有的 frontmatter：
```yaml
type: source|entity|concept|domain|comparison|question|overview|meta
title: "人类可读标题"
created: 2026-04-07
updated: 2026-04-07
tags: [领域标签, 类型标签]
status: seed|developing|mature|evergreen
related: ["[[其他页面]]"]
sources: ["[[.raw/articles/source.md]]"]
```

### 独特优势
- **Hot Cache** — 每次新 session 自动加载上下文（其他项目没有）
- **Canvas 可视化** — JSON Canvas 文件做可视关系图
- **矛盾标记** — [!contradiction] callout 双向标注冲突
- **方法论切换** — 一键切换 LYT/PARA/Zettelkasten
- **DragonScale Memory** — 可选的日志折叠 + 确定性 ID + 语义去重 + 前沿研究
- **多写入安全** — 文件级 advisory lock
- **/think 思考框架** — 10 原则结构化思考
- **隐私优先** — 数据外发需明确同意

## 四、garrytan/gbrain 概览

### 基本信息
- **仓库**: https://github.com/garrytan/gbrain
- **Stars**: ~14,000（2026-04-05 发布）
- **协议**: MIT
- **定位**: OpenClaw/Hermes Agent 的"有态度的大脑"

### 三层架构
1. **Brain Repo** — Git 版本化的 Markdown 文件
2. **GBrain Retrieval** — Postgres 混合检索（BM25 + 向量嵌入）
3. **MCP Server** — Agent 的查询/写入接口

### 核心特色
- Markdown-first 知识图谱：plain-text → 自动连线
- `gbrain features --json --auto-fix`：自动诊断未使用功能并修复
- 健康检查：dead links、missing embeddings
- 对比评测：[GBrain vs Hindsight vs Mem0 vs Zep](https://vectorize.io/articles/gbrain-vs-hindsight-vs-mem0-vs-zep)
- 产品化较深，绑定 OpenClaw 生态

## 六、对 Stitch 的具体借鉴建议

### 从 nashsu/llm_wiki 借鉴：
1. **两步链式思考 Ingest** — 分析→生成，质量显著更高
2. **四信号知识图谱评分模型** — 比纯 wikilinks 更智能
3. **purpose.md 概念** — "Wiki 的灵魂"，定义目标和研究范围
4. **Deep Research 流程** — LLM 主动搜索填补知识空白
5. **上下文预算控制** — 60/20/5/15 比例分配
6. **SHA256 增量缓存** — 跳过未变化文件
7. **Chrome Web Clipper** — 浏览器直接 ingest

### 从 claude-obsidian 借鉴：
1. **Hot Cache** — 500 词近期上下文，新 session 秒加载
2. **[!contradiction] callout** — 矛盾双向标注机制
3. **10 类 Lint 检查** — 最完整的健康维护体系
4. **方法论模式路由** — 同一系统支持多种组织哲学
5. **Skill 分层设计** — 15 个独立 skill 的模块化架构
6. **Hook 体系** — SessionStart/Stop/PostToolUse/PostCompact 四钩子
7. **Auto-commit** — wiki 变更自动 git 提交

### 从 gbrain 借鉴：
1. **三层分离** — Repo(存储) / Retrieval(检索) / Server(接口)
2. **Postgres 混合检索** — 当规模超过 qmd 能力时的升级路径
3. **Git-versioned brain** — 知识库即 git 仓库的理念
4. **gbrain features 自诊断** — 自动发现未使用功能

## 七、最终推荐组合

```
Stitch = nashsu 的编译引擎
       + claude-obsidian 的维护机制
       + gbrain 的存储理念
```

具体来说：
- **Ingest 核心**: nashsu 的两步链式 + SHA256 增量
- **Wiki 维护**: claude-obsidian 的 Hot Cache + 10 类 Lint + 矛盾标记
- **检索层**: qmd v3（轻量）→ 规模大了升级 gbrain Postgres
- **存储**: Git-versioned Markdown（gbrain 理念）
- **Obsidian 适配**: claude-obsidian 的 Schema + 方法论模式
- **主动研究**: nashsu 的 Deep Research 流程
- **可视化**: nashsu 的 sigma.js 四信号图谱
